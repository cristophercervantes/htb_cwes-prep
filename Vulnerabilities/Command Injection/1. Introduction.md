# Command Injection 

Injection happens when user-controlled input gets interpreted as part of a query or command rather than just data. The application thinks it's passing a value somewhere; the attacker makes it pass instructions instead.

OS command injection is one flavor of this. Others you'll run into:

| Type                 | Where it fires                                   |
| -------------------- | ------------------------------------------------ |
| OS Command Injection | User input lands inside a shell command          |
| Code Injection       | User input lands inside an `eval`-style function |
| SQL Injection        | User input lands inside a database query         |
| XSS / HTML Injection | User input gets rendered directly in the page    |

There are more (LDAP, NoSQL, XPath, HTTP header, ORM...) — the pattern is the same across all of them. Input goes into a query. If it's not properly separated from the query structure, the input can rewrite the query.

## OS Command Injection

The specific condition: user input reaches a function that executes shell commands, without being sanitized first.

### PHP

PHP has several functions that hand strings directly to the OS: `exec`, `system`, `shell_exec`, `passthru`, `popen`. Any of them with unsanitized input is a problem.

```php
<?php
if (isset($_GET['filename'])) {
    system("touch /tmp/" . $_GET['filename'] . ".pdf");
}
?>
```

The app intends to create a PDF with a user-supplied name. But the `filename` parameter goes straight into `system()` with no sanitization. Anything you put in there runs as a shell command.

### Node.js

Same idea, different functions — `child_process.exec` and `child_process.spawn`:

```javascript
app.get("/createfile", function(req, res){
    child_process.exec(`touch /tmp/${req.query.filename}.txt`);
})
```

The `filename` query parameter is interpolated directly into a template string that becomes a shell command. Identical vulnerability, identical exploitation.

The language doesn't matter — PHP, Node, Python, Ruby, Java all have ways to shell out, and all of them are vulnerable if they pass user input through without sanitizing it. Command injection also isn't limited to web apps — any binary or thick client that runs shell commands with user-controlled input has the same exposure.
