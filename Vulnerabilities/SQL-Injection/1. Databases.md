# Intro to Databases

Early web apps used flat file-based databases. Fine when small, increasingly painful as data grew — slow, hard to manage, didn't scale. That led to **Database Management Systems (DBMS)**: software specifically designed to create, host, and manage databases efficiently.

Types of DBMS that exist:

- **Relational (RDBMS)** — the classic, SQL-based kind (MySQL, PostgreSQL, MSSQL)
- **NoSQL** — document stores, key/value stores (MongoDB, Redis)
- **Graph-based** — for relationship-heavy data (Neo4j)
- **Key/Value stores** — fast lookups, often used for caching (Redis, DynamoDB)

For SQL injection purposes, we're mostly focused on **relational databases**.

## What a DBMS actually does 

|Feature|Why it matters|
|---|---|
|**Concurrency**|Multiple users hitting the app at the same time without corrupting each other's data|
|**Consistency**|Ensures data stays valid even during simultaneous writes|
|**Security**|Fine-grained access control — who can read what, who can write what|
|**Reliability**|Easy backups, rollbacks if something goes wrong|
|**SQL**|A standardized, human-readable language to interact with all of it|

## The three-tier architecture

Most web apps follow this structure:

```
[User / Browser]
      ↓  (clicks, form submissions)
[Tier I — Client-side app]
      ↓  (API calls, HTTP requests)
[Tier II — Application server / middleware]
      ↓  (SQL queries)
[Tier III — DBMS]
```

**Tier I** — the frontend. What the user actually sees and interacts with. Websites, mobile apps, GUIs.

**Tier II** — the middleware / application server. This is where the backend logic lives. It takes user inputs, figures out what database operation is needed, and sends the right query. This is also where _a lot_ of vulnerabilities live — including SQL injection, which happens when user input from Tier I gets improperly handled by Tier II before being sent to the DBMS.

**Tier III** — the DBMS itself. Receives queries, runs them, returns data (or errors). It trusts that whoever is sending it queries has already validated the input. Spoiler: sometimes they haven't.

> The important thing to internalize: the DBMS just executes what it's told. If bad input makes it into a query, the DBMS runs it anyway. That's the root of SQL injection.

# Types of Databases 
## Relational Databases (SQL)

The most common type. Built around **tables** — rows and columns, like a spreadsheet. Every table has a defined structure (schema) and tables relate to each other through **keys**.

Classic example: a web app might have a `users` table and a `posts` table.

```
users                          posts
─────────────────────          ──────────────────────────────
id | username | first_name     id | user_id | date | content
─────────────────────          ──────────────────────────────
1  | jsmith   | John           1  |    1    | ...  | "Hello"
2  | ajones   | Alice          2  |    2    | ...  | "World"
```

Instead of duplicating all user info in every post, `posts.user_id` just points to `users.id`. Clean, no redundancy, and you can pull everything about a user across every table with one query.

That link between tables is the whole point of _relational_ databases. The schema defines how everything connects.

The schema tells you exactly what data exists and where. If you can query one table, you might be able to reach others. The entire connected structure becomes your attack surface.

Common relational databases: **MySQL, PostgreSQL, MSSQL, Oracle, SQLite**

## Non-Relational Databases (NoSQL)

No tables. No fixed schema. No SQL. Instead, data is stored in flexible formats depending on the model.

Four main storage models:

|Model|How it stores data|
|---|---|
|**Key-Value**|Like a dictionary — key maps to a value (string, object, JSON)|
|**Document-Based**|JSON/BSON documents, each self-contained (MongoDB)|
|**Wide-Column**|Like a table but columns can vary per row|
|**Graph**|Nodes and edges — great for relationship-heavy data|

Key-Value example in JSON:

```json
{
  "100001": {
    "date": "01-01-2021",
    "content": "Welcome to this web application."
  },
  "100002": {
    "date": "02-01-2021",
    "content": "This is the first post on this web app."
  }
}
```

Looks just like a Python dict or a JS object. No rigid structure — you can add whatever fields you want to each entry.

**Why NoSQL exists:** when your data doesn't fit neatly into rows and columns, or when you need to scale massively and flexibly, NoSQL works better than forcing everything into a rigid schema.

Most common example: **MongoDB**
