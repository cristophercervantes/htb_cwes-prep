# Validating Fuzzing Findings 

Because false positives are real and annoying. Your fuzzer flagged something — great. But before you report it or try to exploit it, you need to confirm:

- Is this actually a vulnerability, or just a weird server quirk?
- How bad is it if it _is_ real? What's the actual impact?
- Can you reproduce it consistently?
- Do you have enough evidence to show someone else?

Skipping validation means you might waste time chasing ghosts, or worse — confidently report something that isn't real.

## How to validate — the manual approach

The most reliable method is just doing it yourself. Three steps:

1. **Reproduce the request** — use `curl` or your browser to manually send the same request that triggered the finding
2. **Analyze the response** — look for error messages, unexpected content, anything that shouldn't be there
3. **Exploit (carefully)** — if it looks real, try to confirm the impact with a safe proof of concept

That last step needs some emphasis. The goal is a **PoC that proves the vulnerability exists without actually causing damage**. For example, if you suspect SQL injection, craft a query that returns the SQL server version — not one that drops tables or extracts real user data. Prove it's there, don't make it worse.

> Only do exploitation steps with proper authorization. This is non-negotiable.

## Worked example — finding a `/backup/` directory

Say your fuzzer found `/backup/` and got a `200 OK`. That's worth investigating. Backup directories are goldmines for attackers because they often contain:

- **Database dumps** — full user tables, credentials, all of it
- **Config files** — API keys, encryption keys, environment variables
- **Source code** — which might reveal other vulnerabilities

First step: confirm the directory is actually browsable.

```bash
curl http://IP:PORT/backup/
```

If the server returns an HTML directory listing with files in it, that's your confirmation. Directory listing is enabled and the folder is accessible. You might see something like:

```
backup.sql   2024-Jun-12 14:00:46   0.2K
```

Okay so there's a `.sql` file sitting in a publicly accessible backup directory. That's bad. But here's the responsible part — **don't just download it**.

## The responsible way to confirm — check headers only

Instead of fetching the file and reading potentially sensitive data, just grab the headers:

```bash
curl -I http://IP:PORT/backup/password.txt
```

You'll get something like:

```
HTTP/1.1 200 OK
Content-Type: text/plain;charset=utf-8
Content-Length: 171
Last-Modified: Wed, 12 Jun 2024 14:08:46 GMT
Server: lighttpd/1.4.76
```

What this tells you without reading the file:

- **`Content-Type: text/plain`** — it's a plain text file (makes sense for a password file)
- **`Content-Length: 171`** — the file isn't empty. 171 bytes of _something_ is in there.
- **`200 OK`** — it's publicly accessible, no auth required

A file called `password.txt` in a backup folder with 171 bytes of content, accessible to anyone? That's more than enough to call this a confirmed vulnerability. You've proven it exists and assessed the risk without ever reading the actual passwords.

> If `Content-Length` is 0, pump the brakes a bit. An empty file is suspicious but not necessarily a direct risk. Check the filename and context before escalating.

## curl flags worth knowing for validation

|Command|What it does|
|---|---|
|`curl http://target/path/`|Fetch the full response — confirms directory listing, content, etc.|
|`curl -I http://target/file`|Fetch headers only — safer for sensitive files|
|`curl -v http://target/path/`|Verbose — shows request and response headers together|
|`curl -o /dev/null -s -w "%{http_code}" http://target/`|Just get the status code, nothing else|
