**Directory & File Fuzzing**

Web apps often have directories and files that aren't linked anywhere — hidden from normal users. These can contain backup files, config files, old vulnerable scripts, admin panels, or dev/staging environments. Finding them is a big part of any pentest.

**Wordlists**

Fuzzing tools like ffuf don't come with built-in wordlists — you bring your own. The go-to collection is **SecLists** (github.com/danielmiessler/SecLists). On Pwnbox it's at `/usr/share/seclists/`.

The ones you'll actually use:

|Wordlist|When to use it|
|---|---|
|`Discovery/Web-Content/common.txt`|Default starting point|
|`Discovery/Web-Content/directory-list-2.3-medium.txt`|Deeper directory hunting|
|`Discovery/Web-Content/big.txt`|Cast a wide net|

**How ffuf works**

Simple idea — you give it a wordlist and a URL with `FUZZ` as a placeholder. It replaces `FUZZ` with every word in the list and checks the response.

**Directory fuzzing**

```bash
ffuf -w /usr/share/seclists/Discovery/Web-Content/directory-list-2.3-medium.txt -u http://IP:PORT/FUZZ
```

A `301` response means the directory exists. Here it found `/w2ksvrus` — worth digging into.

**File fuzzing**

Once you find a directory, go deeper and look for specific files. Use `-e` to append extensions to every word in the list:
```bash
ffuf -w /usr/share/seclists/Discovery/Web-Content/common.txt -u http://IP:PORT/w2ksvrus/FUZZ -e .php,.html,.txt,.bak,.js -v
```

- `.php` — server-side scripts, possible logic flaws
- `.bak` — backup files, often contain credentials
- `.txt` — logs or notes left by devs
- `.js` — client-side code, sometimes leaks API keys

This found two files inside `/w2ksvrus`:

- `index.html` — probably the default page
- `dblclk.html` — unknown purpose, investigate further

**Recursive Fuzzing**

So far we've only fuzzed one level at a time — root directory, then manually go into a found directory and fuzz again. That gets tedious fast on a real target. Recursive fuzzing automates this completely.

**How it works**

It's basically a tree search:

1. Fuzz the root, find a directory (e.g. `/admin`)
2. Automatically start fuzzing inside that directory (`/admin/FUZZ`)
3. If it finds another directory inside, fuzz that too
4. Keeps going until it hits your depth limit or runs out of directories

**The command**


```bash
ffuf -w /usr/share/seclists/Discovery/Web-Content/directory-list-2.3-medium.txt -ic -v -u http://IP:PORT/FUZZ -e .html -recursion
```

 flags:

- `-recursion` — enables recursive mode
- `-ic` — ignores comment lines in the wordlist (lines starting with `#`)
- `-v` — verbose, shows full URLs as they're found

**What the output looks like**

ffuf found `/level1` (301 redirect) → automatically queued `/level1/FUZZ` Inside level1 it found `/level2` and `/level3` → both queued automatically Then found `index.html` in each one

The `index.html` in `level3` was slightly larger than the others — worth checking. Turned out to contain the flag.

That's the key habit: **pay attention to file size differences** in results. Same filename but different size = something interesting inside.

**Be careful though**

Recursive fuzzing hammers the server with requests. On a real engagement this can cause performance issues or trigger alerts. Always throttle it:


```bash
ffuf -w /usr/share/seclists/Discovery/Web-Content/directory-list-2.3-medium.txt -ic -u http://IP:PORT/FUZZ -e .html -recursion -recursion-depth 2 -rate 500
```

- `-recursion-depth 2` — don't go deeper than 2 levels
- `-rate 500` — max 500 requests per second
- `-timeout` — set a timeout so it doesn't hang on slow responses

In most real scenarios, depth 2-3 is enough. Going deeper just creates noise.
