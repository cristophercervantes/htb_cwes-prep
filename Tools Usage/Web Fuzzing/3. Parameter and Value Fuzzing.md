## Parameter & Value Fuzzing

When you visit a website, your browser and the server are constantly passing information back and forth. **Parameters** are how that information gets passed — they're basically labeled boxes carrying data. Fuzzing means we're going to throw thousands of different values at those boxes to see what breaks or behaves unexpectedly.

### Two Types of Parameters

#### GET Parameters — Out in the Open

These live right in the URL where anyone can see them:

```
https://example.com/search?query=fuzzing&category=security
```

The `?` starts the parameters, and `&` separates multiple ones. Think of it like a postcard — anyone who glances at it can read everything on it. GET parameters are fine for harmless stuff like searches and filters.

#### POST Parameters — Hidden in the Envelope

These are sent inside the request body, completely invisible in the URL. This is why login forms use POST — you don't want your password showing up in the browser bar for everyone to see.

A POST request looks like this under the hood:

```
POST /login HTTP/1.1
Host: example.com
Content-Type: application/x-www-form-urlencoded

username=john&password=secret123
```

The data travels hidden inside the body of the request, not the URL.

> Two common formats for POST data: `application/x-www-form-urlencoded` (regular form data, looks like GET but in the body) and `multipart/form-data` (used when uploading files).

Parameters are essentially the doors into a web application. If the app doesn't properly check what values come in, bad things can happen:

- Change a product ID in a shopping cart URL → maybe you can see someone else's order, or get things for free
- Tweak a hidden parameter → maybe you unlock admin features
- Inject code into a search box → XSS or SQL injection

The goal of parameter fuzzing is to find the values that make the app do something it shouldn't.

### Fuzzing GET Parameters with `wenum`

#### Step 1 — Install wenum


```bash
pipx install git+https://github.com/WebFuzzForge/wenum
pipx runpip wenum install setuptools
```

#### Step 2 — Poke the endpoint manually first

Before throwing a wordlist at it, just check what the server says normally:


```bash
curl http://IP:PORT/get.php
# Response: Invalid parameter value  x:
```

The server is telling us: _"Hey, I expect a parameter called x, but you didn't give me one."_

Try giving it a value:


```bash
curl http://IP:PORT/get.php?x=1
# Response: Invalid parameter value  x: 1
```

Now it sees the parameter but says the value `1` is wrong. So the server is definitely checking values — we just need to find the right one.

#### Step 3 — Let wenum do the heavy lifting

Instead of guessing by hand (which could take forever), we automate it:


```bash
wenum -w /usr/share/seclists/Discovery/Web-Content/common.txt \
      --hc 404 \
      -u "http://IP:PORT/get.php?x=FUZZ"
```

What each part means:

- `-w` → the wordlist to try (thousands of common words)
- `--hc 404` → hide 404 responses so the output isn't cluttered
- `FUZZ` → this is the placeholder; wenum replaces it with each word from the list one by one

#### Step 4 — Spot the winner

Out of 4,731 requests, almost all come back saying "invalid." One stands out:

```
200   1 L   1 W   25 B   GET   http://IP:PORT/get.php?x=OA...
```

A `200` status code means _success_ — that value works. Confirm it:


```bash
curl http://IP:PORT/get.php?x=OA...
# HTB{...}  ← flag!
```

### Fuzzing POST Parameters with `ffuf`

POST fuzzing is almost the same idea, just the data goes in the request body instead of the URL.

#### Step 1 — Check the endpoint manually


```bash
curl -d "" http://IP:PORT/post.php
# Response: Invalid parameter value  y:
```

The `-d` flag tells curl to send a POST request. The server expects a parameter called `y`.

#### Step 2 — Automate with ffuf

```bash
ffuf -u http://IP:PORT/post.php \
     -X POST \
     -H "Content-Type: application/x-www-form-urlencoded" \
     -d "y=FUZZ" \
     -w /usr/share/seclists/Discovery/Web-Content/common.txt \
     -mc 200 -v
```

Breaking it down:

- `-X POST` → use POST method
- `-H "Content-Type: ..."` → tell the server we're sending form data
- `-d "y=FUZZ"` → the POST body; ffuf replaces FUZZ with each word
- `-mc 200` → only show responses with status code 200 (the successes)

#### Step 3 — Find the hit

```
[Status: 200]  FUZZ: SU...
```

Confirm it:

```bash
curl -d "y=SU..." http://IP:PORT/post.php
# HTB{...}  ← flag!
```
