# Web APIs 

> An API is basically a middleman. One piece of software wants something from another piece of software, and the API is the agreed-upon way they communicate. Think of it like a waiter — you don't walk into the kitchen yourself, you tell the waiter what you want and they handle the rest.

## The three main types

### REST (Representational State Transfer)

The most common one you'll run into. Uses standard HTTP methods to do things with resources:

- `GET` → read something
- `POST` → create something
- `PUT` → update something
- `DELETE` → delete something

Each resource has its own URL, data comes back as JSON (usually), and the whole thing is stateless — meaning each request is independent, the server doesn't remember previous requests.

```
GET /users/123
```

Simple, readable, and works with basically everything. That's why it's so popular.

### SOAP (Simple Object Access Protocol)

Older, more formal, more verbose. Everything is XML wrapped in a "SOAP envelope." Used a lot in enterprise environments where you need strict error handling, security guarantees, and transaction management.

```xml
<soapenv:Envelope>
  <soapenv:Body>
    <tem:GetStockPrice>
      <tem:StockName>AAPL</tem:StockName>
    </tem:GetStockPrice>
  </soapenv:Body>
</soapenv:Envelope>
```

You can immediately see why REST won for most use cases. That said, SOAP is still alive in banking, healthcare, and legacy enterprise systems — places where reliability matters more than convenience.

### GraphQL

The newer one. Instead of having a different endpoint for every resource (like REST does), GraphQL gives you _one_ endpoint and lets you ask exactly for what you need.

```graphql
query {
  user(id: 123) {
    name
    email
  }
}
```

This solves two REST problems:

- **Over-fetching** — getting back a huge object when you only needed one field
- **Under-fetching** — having to make multiple requests to get everything you need

The tradeoff is it's more complex to set up and has its own attack surface (introspection queries, for instance, which is relevant for fuzzing later).

## APIs vs. regular web servers 

This matters a lot for fuzzing, so worth being clear on:

| Web Server | API |
|------------|-----|
| **What it serves** | HTML, CSS, images — stuff for browsers | Data (JSON, XML) — stuff for other apps |
| **Who interacts with it** | Humans, through a browser | Other applications, on behalf of users |
| **Data format** | Web formats | JSON, XML, etc. |
| **Endpoints** | Pages and directories | Resources and actions |
| **Access** | Usually public | Public, private, or partner-only |

The practical takeaway: **when you're fuzzing an API, you're not looking for hidden directories or files — you're fuzzing endpoints and parameters.** The data formats matter more. The HTTP methods matter more. It's a different mindset than fuzzing a regular web server.

# Identifying API Endpoints

> Before you can fuzz an API, you need to know what to fuzz. That sounds obvious, but APIs aren't always self-documenting and endpoints aren't always visible. This is the recon phase.

The approach varies a lot depending on which type of API you're dealing with — REST, SOAP, or GraphQL each have their own quirks.

## REST APIs

REST is built around _resources_ with unique URLs. The structure is usually pretty readable:

```
/users              → collection of all users
/users/123          → one specific user
/products           → collection of products
/products/456       → one specific product
```

Hierarchical, logical, human-readable. That's the idea anyway.

### Parameter types to know

|Type|Where it lives|What it's for|Example|
|---|---|---|---|
|Query params|After `?` in the URL|Filtering, sorting, pagination|`/users?limit=10&sort=name`|
|Path params|Embedded in the URL|Identifying a specific resource|`/products/{id}`|
|Body params|In the request body|Creating or updating things|`{ "name": "New Product", "price": 99.99 }`|

### How to discover REST endpoints

- **Official docs** — check first, always. Look for Swagger/OpenAPI specs, they're machine-readable and list everything. If there's a `/swagger.json` or `/openapi.yaml` floating around, that's a goldmine.
- **Browser dev tools / Burp Suite** — just use the app normally and watch the network tab. Every API call the frontend makes is an endpoint you now know about.
- **Parameter name fuzzing** — same tools as directory fuzzing (ffuf, wfuzz) but pointed at parameter names. Useful for finding hidden or undocumented params the docs don't mention.

## SOAP APIs

SOAP is different. Instead of multiple endpoints for different resources, there's usually **one URL** that handles everything. The actual operation is determined by what's inside the XML message body.

```xml
<soapenv:Envelope>
  <soapenv:Body>
    <lib:SearchBooks>
      <lib:keywords>cybersecurity</lib:keywords>
      <lib:author>Dan Kaminsky</lib:author>
    </lib:SearchBooks>
  </soapenv:Body>
</soapenv:Envelope>
```

The "endpoint" here is `SearchBooks` — but you're not visiting `/SearchBooks` as a URL, you're sending it as part of the XML payload to one endpoint.

### How to discover SOAP operations

- **WSDL file** — this is _the_ resource for SOAP. WSDL (Web Services Description Language) describes every operation, every input parameter, every output format, and the data types involved. If you can get the WSDL, you basically have the full API spec. Try appending `?wsdl` to the endpoint URL.
- **Network traffic** — Wireshark or tcpdump to capture SOAP traffic if you can get in a position to observe it
- **Fuzzing** — even with a well-defined structure, sending unexpected values inside SOAP requests can reveal undocumented operations or how the server handles bad input

## GraphQL

GraphQL is the most interesting one from a recon perspective. Like SOAP, it usually has **a single endpoint** (typically `/graphql`). But unlike SOAP's rigid XML, GraphQL uses its own query language that's much more flexible.

Two types of operations:

### Queries (reading data)

```graphql
query {
  user(id: 123) {
    name
    email
    posts(limit: 5) {
      title
      body
    }
  }
}
```

You ask for exactly what you need — specific fields, nested relationships, with arguments to filter or paginate. No over-fetching, no under-fetching.

### Mutations (writing data)

```graphql
mutation {
  createPost(title: "New Post", body: "Content here") {
    id
    title
  }
}
```

Same idea, but for creating/updating/deleting. You also specify what you want back in the response after the mutation runs.

### How to discover GraphQL schema

- **Introspection** — this is huge. GraphQL has a built-in introspection system. Send an introspection query to the endpoint and you get back the _entire schema_ — every type, every field, every query, every mutation, every argument. It's like the API describing itself to you. Some servers disable it in production, but many don't.
- **GraphiQL / GraphQL Playground** — if these are enabled (sometimes they are on dev or staging), you get an interactive UI to explore the schema and write queries. Super useful.
- **Network traffic** — same as always — watch what the frontend sends to `/graphql` and you'll see real queries in the wild

> Introspection being enabled is itself worth noting as a finding. It exposes the full API surface to anyone who knows to ask.

# API Fuzzing

When you fuzz an API, you're not just looking for hidden directories. You're messing with:

- **Parameter values** — swap in weird types, out-of-range numbers, empty strings, SQL fragments
- **Request headers** — content-type, auth headers, custom headers
- **Parameter order** — some APIs behave differently depending on how you structure the request
- **Data formats** — malformed JSON, broken XML, unexpected encoding

The goal is to trigger errors, crashes, or behavior the API wasn't supposed to expose — which usually points to something interesting underneath.

- APIs often have **hidden or undocumented endpoints** — stuff that didn't make it into the docs but is still live on the server
- Developers assume the frontend will send "normal" requests — the API may not handle weird input gracefully
- Manual testing every possible input combination is completely infeasible, fuzzing automates it
- You're essentially simulating what a real attacker would do — find that stuff before they do

## The three types of API fuzzing

### 1. Parameter Fuzzing

Testing different values for query params, headers, and request bodies. This is the most common type. You're looking for things like:

- SQL injection
- Command injection
- XSS
- Parameter tampering (changing `user_id=123` to `user_id=456` and seeing if you get someone else's data)

### 2. Data Format Fuzzing

APIs exchange data in JSON, XML, etc. Data format fuzzing goes after the _structure_ of that data — malformed JSON, nested objects where they shouldn't be, weird character encodings, oversized strings. Reveals parsing errors, buffer overflows, improper handling of special characters.

### 3. Sequence Fuzzing

APIs often have workflows — you create something, then you read it, then you update it. Sequence fuzzing messes with _the order and timing_ of those calls. This is where you catch things like:

- Race conditions
- IDOR (Insecure Direct Object References)
- Authorization bypasses (does step 3 actually check if you completed step 2?)

## A real example — finding a hidden endpoint

Say an API has documented endpoints in its Swagger UI (`/docs`):

```
GET  /           → root, probably just a welcome message
GET  /items/{id} → fetch a specific item
DELETE /items/{id}
PUT  /items/{id}
POST /items/
```

Five endpoints, all documented. But documentation doesn't mean _complete_ documentation. There could be more.

Run a fuzzer against it:

```bash
git clone https://github.com/PandaSt0rm/webfuzz_api.git
cd webfuzz_api
pip3 install -r requirements.txt
python3 api_fuzzer.py http://IP:PORT
```

Output:

```
Status code counts:
  404: 4727
  200: 2
  405: 1

Found valid endpoints:
  - http://localhost:8000/cz...   ← undocumented, not in the docs
  - http://localhost:8000/docs    ← expected

Unusual status codes:
  405: http://localhost:8000/items
```

A few things to notice here:

- `/cz...` returned a 200 but isn't in the docs at all — that's the interesting find
- `/docs` is expected — confirmation the fuzzer is working
- The `405 Method Not Allowed` on `/items` means the endpoint _exists_ but the fuzzer hit it with the wrong HTTP method (probably tried GET instead of POST). Still useful to know.

Hit the undocumented endpoint:

```bash
curl http://localhost:8000/cz...
# → {"flag":"<snip>"}
```

There it is. An endpoint that was never in the docs, sitting there responding to requests.

## What parameter fuzzing can expose

Once you know the endpoints, fuzzing the _parameters_ those endpoints accept can reveal:

- **Broken Object-Level Authorization** — change `item_id=1` to `item_id=2` and get someone else's item
- **Broken Function-Level Authorization** — manipulate params to call functions you shouldn't have access to
- **SSRF (Server-Side Request Forgery)** — inject a URL into a parameter and trick the server into making requests on your behalf — could expose internal services
